## Options

Command-line options are described by the Option record type.
The high-level syntactic interface to creating Options will be
described first, followed by the high-level procedural interface,
and then finally the lower-level core procedures exported by
‘(steinmetz options)’.


### Syntactic interface

(options <clause> ...) → (list Option)

The 'options' form is a convenient interface to the options library.
Each clause describes a single option and is of one of the following
forms:

    (flag   <name-or-names> [<help-text>])
    (option <name-or-names> [<arg-name> [<help-text> [<conv>]]])

The (flag …) form describes a boolean flag which takes no arguments;
(option …) describes an option taking an argument.
<name-or-names> is either a string, an identifier, or a list of
strings or identifiers,
and gives the acceptable forms of an option (with leading dashes
omitted). <arg-name> and <conv> have
the same meaning as they do in make-option. <help-text> is a string
describing the option.


Note: Option names may be written as identifiers for convenience, but
will be stored as strings in the actual Option objects.  For example,

    (map option-names (options (flag (v verbose))))
      ⇒ (("v" "verbose"))


The acceptable command-line form of each name is determined by its
length.  A single-character name like ‘o’ will be parsed as a “short”
option, i.e. ‘-o’, while a two-or-more-character name like ‘output’
will be parsed as a “long” option, i.e. ‘--output’.  See Parsing
conventions below for more details on how options and their arguments
are parsed.

The value of an 'options' form is a list of option structures
representing the CLI options described by its clauses.

### Procedural interface

#### High-level

These procedures are exported by ‘(steinmetz parse)’.  They
automatically generate parsers and fill in details which users of the
‘(steinmetz options)’ interface must provide explicitly.


(make-cli-option names [arg-name [conv-proc [properties]]]) → option

Argument types:

(: names (list string))
(: arg-name (or symbol #f))
(: conv-proc procedure)
(: properties (alist symbol *))

Returns an Option object describing a command-line option.

*names* gives the command-line forms of the option, with leading dashes
omitted.

If *arg-name* is provided, the command-line option will take a (single)
required argument.  If it is false, the option will not take an
argument (and its parser will raise an exception if an argument is
provided).  TODO: Optional arguments.

If *conv-proc* is provided, it must be a procedure which accepts a
single string argument and returns a single value.  *conv-proc* is
applied to the value of the option’s command-line argument.
*conv-proc* defaults to ‘values’.

If *properties* is provided, it must be a symbol-keyed alist.  Any
number of arbitrary properties may be added.  (See also “Common
properties” below.)  *properties* defaults to an empty list.

The parser generated by ‘make-cli-option’ will associate a name of the
option with the value of ‘(*conv-proc* *arg*)’, where *arg* is the
option’s command-line argument, if the option is encountered at all.

(make-cli-flag names [properties]) → option

A restricted version of ‘make-cli-option’ for creating “flags”, options
which take no arguments and are merely present or absent.  *names* and
*properties* have the same types and meanings they do for
‘make-cli-option’.

The parser generated by ‘make-cli-flag’ will associate a name of the
flag with ‘#t’, if the flag is encountered at all.


##### Common properties

Most options include a 'help' property associated with a string
describing the option.


#### Core procedures

(make-option names argument-name argument-parser properties) → option

Argument types:

(: names (list string))
(: argument-name (or symbol #f))
(: argument-parser procedure)
(: properties (alist symbol *))

Returns a new Option with fields determined by the arguments.

*argument-parser* must accept a single string-list argument and return
two values: the value of the option’s argument and a suffix of the list
argument.  An argument parser should signal a parser exception if its
input list is invalid.  For example, the following is a valid parser:

    (lambda (tokens)
      (if (null? tokens)
          (parser-exception "missing argument" tokens)
          (values (car tokens) (cdr tokens))))


(option-names option) → (list string)

Returns a list of the accepted names of *option*.


(option-argument-name option) → (or symbol #f)

Returns the symbolic name of *option*’s argument.


(option-argument-parser option) → procedure

Returns the argument parser associated with *option*.


(option-properties->alist option) → (alist symbol *)

Returns the properties mapping of *option* as a symbol-keyed alist.
Mutating this alist causes undefined behavior: it may or may not be
*option*’s internal representation of its properties mapping.


(option? obj) → boolean

Returns ‘#t’ if *obj* is an Option and ‘#f’ otherwise.


An Option’s property alist can be manipulated with the following
convenience procedures.


(option-property-ref option key) → *

Returns the value associated with *key* (a symbol) in *option*'s
property mapping.  If no association exists, returns #f.  (FIXME:
This should probably raise an exception on failure, to avoid the
ambiguity associated with returning #f.)


(option-set-property option key obj) → option

Returns a copy of *option* with a new property association of
*key* to *obj*.  Any existing association for *key* is replaced.




## Parsing

### Parsing conventions

A single option may have multiple “short” (single-character) and “long”
(two-or-more-character) names.  Following tradition, short options
without arguments followed by a single option that may take an argument
can be run together.  For example, ‘-ab’ is
equivalent to ‘-a -b’ if the -a option takes no argument.  The argument of a short option is always the
next token.[†]  For a long option, the argument is the next token unless
an argument was supplied as part of the option token, with the two
components being separated by an ‘=’ character.  For example,
‘--output=foo.log’ is equivalent to ‘--output foo.log’.  Note that “X
Windows–style” long options with a single leading dash (e.g.
‘-output’) are not supported, since they conflict unavoidably with
short-option clusters.

†: FIXME: Steinmetz should support short options run together with
their arguments, e.g. -ffoo.


### Procedures

(parser-exception message irritant ...)

Raises a continuable exception indicating that something went wrong
during command-line parsing.  *message* must be a string.  The
*irritant*s may be anything.  The raised object is a compound condition
satisfying ‘parser-condition?’ and including (at least) ‘&message’ and
‘&irritants’ components.


(parser-condition? obj) → boolean

True of *obj* is a condition raised by ‘parser-exception’.


(process-command-line options [cl-list]) -> [alist, list]

A easy-to-use high-level parsing procedure.  *options* is a list of
Option objects.  Parses *cl-list* (a list of strings, or the value of
‘(cdr (command-line))’ by default) and returns two values: an alist
associating the name of each option encountered (as a string) with a list of
arguments, and
a list of tokens unassociated with any of the options (the “operands”).
The order of *cl-list* is preserved in the operand list and in each
option’s argument list, but the order of the options-and-arguments
alist is unspecified.

If an unknown option is encountered while parsing, an object satisfying
‘parser-condition?’ is raised continuably.

Example:

    (let ((opts (options
                  (option (f) FILE "input file")
                  (flag (v) "verbose output")))
          (cl '("-f" "file1" "-v" "-f" "file2" "frobnitz")))
      (process-command-line opts cl))

    ⇒ (("v" #t) ("f" "file1" "file2"))
      ("frobnitz")

(parse-command-line options proc cl-list seed ...) → *

Argument types:

(: options (list Option))
(: proc (procedure))
(: cl-list (list string))
(: seed *)

A general-purpose parsing procedure similar to SRFI 37's 'args-fold'.
Folds *proc* over the result of parsing *cl-list* with the parsers
attached to the *options*.  At each step, *proc* is applied to an
Option, the option's argument token (a string, or #f if the
option has no argument), and the current seeds.  For example, if the
first two elements of *cl-list* are "-f" and "foo", then *proc* will
be invoked as ‘(proc f-opt "foo")’, where *f-opt* is the Option whose
names include ‘f’.

The *seed* arguments are used as the initial seeds, and
the values returned by *proc* are used as the seeds for the next step.

If a token is encountered which is not an option or an option’s
argument, it is passed as the second argument to *proc*, and *proc*'s
first (option) argument will be #f.  For example, if the first element
of *cl-list* is "frob", then *proc* will be invoked as ‘(proc #f
"frob")’.

*parse-command-line* returns the values returned by the final
invocation of *proc*.
